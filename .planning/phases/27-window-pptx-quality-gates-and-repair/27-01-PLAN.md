# Phase 27 Plan 01: Five-Layer QA and Bounded Repair

## Objective

Replace placeholder hooks with stable five-layer quality reports, customer-delivery hard gates, and reversible candidate-only repair that can be proven monotonic.

## Work Packages

- [x] Add failing tests and JSON schemas for quality reports, findings, hard gates, snapshots, and repair logs.
- [x] Inspect package/transaction, COM/editability, geometry/text capacity, visual density/compatibility, and deck consistency.
- [x] Detect overflow risk, tiny type, boundary drift, overlap drift, unreadable chart/table labels, repeated layouts, rasterized slides, missing native objects, and compatibility defects.
- [x] Define stable severity weights and customer-delivery hard-gate codes.
- [x] Implement reversible candidate-only repairs for registered structural defects.
- [x] Limit repair to two passes and accept a pass only when score decreases without hard-gate regression; otherwise roll it back.
- [x] Finalize package/reopen/source-integrity evidence from the existing candidate transaction.
- [x] Integrate report/log serialization and atomic audit artifacts into `PipelineResult` and CLI JSON without breaking custom hooks.
- [x] Run focused and complete suites plus repeated read-only OpenCode audit attempts, and record all evidence for the five QA requirements.

## Exit

Phase 28 remains blocked if reports are unstable, visual/deck checks are absent, native/editability hard gates can be bypassed, repair edits a source or final output, repair can loop indefinitely, a worse pass can be accepted, or transaction evidence is not reflected in the final quality report.

All implementation exit conditions pass. The provider-side OpenCode sessions read the bounded audit ranges but failed to emit a textual verdict; this infrastructure limitation is recorded in validation and the mandatory final independent verdict remains gated by `V5-UAT-04` in Phase 29.
