# Phase 27 Plan 01: Five-Layer QA and Bounded Repair

## Objective

Replace placeholder hooks with stable five-layer quality reports, customer-delivery hard gates, and reversible candidate-only repair that can be proven monotonic.

## Work Packages

- [ ] Add failing tests and JSON schemas for quality reports, findings, hard gates, snapshots, and repair logs.
- [ ] Inspect package/transaction, COM/editability, geometry/text capacity, visual density/compatibility, and deck consistency.
- [ ] Detect overflow risk, tiny type, boundary drift, overlap drift, image distortion, unreadable chart/table labels, repeated layouts, style inconsistency, rasterized slides, missing native objects, and compatibility defects.
- [ ] Define stable severity weights and customer-delivery hard-gate codes.
- [ ] Implement reversible candidate-only repairs for registered structural defects.
- [ ] Limit repair to two passes and accept a pass only when score decreases without hard-gate regression; otherwise roll it back.
- [ ] Finalize package/reopen/source-integrity evidence from the existing candidate transaction.
- [ ] Integrate report/log serialization into `PipelineResult` and CLI JSON without breaking custom hooks.
- [ ] Run focused and complete suites, independent OpenCode review, and close only with evidence for all five QA requirements.

## Exit

Phase 28 remains blocked if reports are unstable, visual/deck checks are absent, native/editability hard gates can be bypassed, repair edits a source or final output, repair can loop indefinitely, a worse pass can be accepted, or transaction evidence is not reflected in the final quality report.
