# Phase 25 Plan 01: Deterministic Core Rendering Pipeline

## Objective

Turn governed compiled slides into editable PowerPoint core objects through a pure render-plan boundary, a narrow COM renderer, a recording fake-COM implementation, and the existing transactional output system.

## Work Packages

- [x] Add failing tests for pure render-plan validation, deterministic theme/layout joining, stable object naming, component allocation, and unsupported-content findings.
- [x] Add failing tests for inch-to-point conversion and identical relative geometry across 16:9, 4:3, portrait, and custom page sizes.
- [x] Implement native editable text, basic shapes, governed image frames, slide/master background, footer, grouping, and explicit z-order operations.
- [x] Implement a recording fake-COM object model and assert complete compiler-to-renderer call ordering without Windows.
- [x] Add a runner lifecycle for validate, compile, plan, render, inspect hook, repair hook, and transactional save, with dry-run remaining side-effect free.
- [x] Extend the CLI with explicit DeckPlan compile/render routes without weakening existing terminal safety actions.
- [x] Validate fake-COM editability metadata, deterministic repeats, error context, cleanup, and candidate-only output behavior.
- [x] Run focused and complete suites, obtain independent review, and close only with evidence for all five renderer requirements.

## Exit

Phase 26 remains blocked if the renderer accepts raw model geometry, uses fixed pixels, writes final outputs directly, emits rasterized text, has unstable object order/names, lacks master/footer/group/z-order behavior, cannot run through recording fake COM, or bypasses the established transaction and cleanup gates.
